# StudyGuard 智能自习室座位管理终端项目计划

## 1. 项目名称

**StudyGuard 智能自习室座位管理终端**

基于 **U1P + C2 + U2P + C2 协同架构** 的自习室座位占用、环境监测与手机端管理系统。

---

## 2. 项目背景

在图书馆、自习室等公共学习场景中，经常存在座位状态不透明、临时离开难以判断、长时间占座无人管理、管理员无法及时处理异常座位等问题。传统座位检测方案通常只判断“有人”或“无人”，功能比较单一，难以区分正常使用、临时离开、超时占座和环境异常等情况。

本项目设计一个智能座位管理终端，使用 **U1P 负责底层传感器采集**，使用 **U2P 负责状态判断、界面显示和 HTTP 管理服务**，并通过两块 **C2 ZigBee 子板**完成 U1P 与 U2P 之间的无线数据传输。系统能够采集人体感应、刷卡信息和温湿度数据，在 E5 触摸屏上显示座位状态，同时允许管理员手机通过网卡访问 U2P 的 HTTP 页面，对座位进行查看和管理。

相比原先只使用 U1P 或仅依赖有线串口通信的方案，本方案加入了 C2 无线通信和手机端 HTTP 管理功能，使系统更接近实际物联网场景中的“分布式采集 + 集中管理”模式。U1P 可以放置在座位端进行传感器采集，U2P 可以作为管理终端进行集中显示和网络服务，整体结构更加清晰，也更适合展示系统的完整性。

---

## 3. 设计目标

本系统需要完成以下目标：

1. 使用 U1P、U2P 和两块 C2 子板构成协同控制架构。
2. 使用 S7 人体感应模块判断座位前是否有人。
3. 使用 S5 NFC/RFID 模块模拟学生刷卡预约、签到和签退。
4. 使用 S8 温湿度传感器采集自习室环境数据。
5. 使用 C2 ZigBee 子板实现 U1P 与 U2P 之间的无线数据传输。
6. 使用 E5 HDMI 触摸屏显示座位状态、学生刷卡信息和环境数据。
7. 在 U2P 上运行 HTTP 服务，使管理员手机能够通过浏览器访问座位管理页面。
8. 实现空闲、预约、使用中、临时离开、疑似占座和环境异常等状态判断。
9. 支持管理员通过手机端清除疑似占座状态或强制释放异常座位。

## 4. 硬件选型

| 槽位/模块 | 硬件 | 主要作用 |
|---|---|---|
| #0 | U2P + C2 | 上层状态管理、界面显示、HTTP 服务、ZigBee 接收 |
| #1 | U1P + C2 | 底层传感器采集、ZigBee 发送 |
| #2 | S7 人体感应模块 | 检测座位前是否有人 |
| #3 | S5 NFC/RFID 模块 | 模拟学生刷卡、管理员刷卡 |
| #4 | S8 温湿度传感器 | 采集环境温度和湿度 |
| #hdmi | E5 HDMI 触摸屏 | 本地显示座位地图和管理界面 |
| U2P 网卡 | 有线网卡或无线网卡 | 提供 HTTP 管理页面访问入口 |

其中，C2 子板用于 ZigBee 无线通信。U1P 侧 C2 负责发送座位端采集数据，U2P 侧 C2 负责接收数据并交给 U2P 处理。管理员手机不直接连接 C2，而是通过局域网访问 U2P 提供的 HTTP 服务。

## 5. 系统总体架构

系统采用 **“U1P 底层采集 + C2 无线传输 + U2P 状态管理 + E5 本地显示 + 手机端 HTTP 管理”** 的总体结构。

```text
S7 人体感应  ----\
S5 NFC/RFID  ----- U1P ---- UART ---- C2  ))) ZigBee (((  C2 ---- UART ---- U2P ---- HDMI ---- E5 触摸屏
S8 温湿度    ----/                                                              |
                                                                                 |
                                                                            网卡 / 局域网
                                                                                 |
                                                                            管理员手机
```

各部分分工如下：

```text
U1P：
1. 初始化 GPIO、I2C、UART 等底层接口。
2. 读取 S7 人体感应信号。
3. 读取 S5 NFC/RFID 刷卡数据。
4. 读取 S8 温湿度数据。
5. 对传感器数据进行基础校验和格式化。
6. 通过 UART 将数据交给本地 C2 子板。

U1P 侧 C2：
1. 接收 U1P 通过 UART 发送的数据。
2. 通过 ZigBee 将座位端数据发送给 U2P 侧 C2。
3. 可按需要配置为终端节点或发送节点。

U2P 侧 C2：
1. 接收 U1P 侧 C2 发来的 ZigBee 数据。
2. 通过 UART 将数据传给 U2P。
3. 可按需要配置为协调器或接收节点。

U2P：
1. 运行 Linux/Ubuntu 系统。
2. 接收 C2 上传的人体感应、刷卡和温湿度数据。
3. 根据刷卡状态、人体感应状态和计时器进行座位状态判断。
4. 在 E5 触摸屏上显示座位地图和环境数据。
5. 运行 HTTP 服务，为管理员手机提供网页管理入口。
6. 处理管理员清除占座、释放座位和查看状态等操作。
```

## 6. C2 子板使用方案

C2 子板集成 ZigBee 无线通信模块，可以通过 UART 串口指令进行配置和通信。本项目中，两块 C2 分别安装在 U1P 和 U2P 上，用于替代原计划中的有线 UART 直连，使座位端数据能够通过无线方式发送到管理端。

### 6.1 C2 工作方式

本项目建议使用两块 C2：

```text
U1P + C2：座位端节点
U2P + C2：管理端节点
```

U1P 侧 C2 主要负责发送数据，U2P 侧 C2 主要负责接收数据。两块 C2 需要配置到同一个 PAN_ID，并选择一致的通信模式。为了降低调试难度，前期可以优先使用 HEX 指令模式完成通信测试，再将测试数据替换为实际的座位数据帧。

### 6.2 C2 与 U 板连接

C2 放置在 U 板上时，应工作在通信模式，通过 UART 与 U 板通信。U1P 与 C2 之间、U2P 与 C2 之间都可以看作本地串口通信关系，区别在于两块 C2 之间通过 ZigBee 完成无线转发。

```text
U1P UART -> U1P 侧 C2 -> ZigBee -> U2P 侧 C2 -> U2P UART
```

### 6.3 C2 初始化建议

C2 初始化阶段建议完成以下工作：

```text
1. 设置通信模式，例如 HEX 模式。
2. 查询设备当前状态，确认设备类型和参数。
3. 配置 U2P 侧 C2 为协调器或接收端。
4. 配置 U1P 侧 C2 为终端或发送端。
5. 设置相同 PAN_ID。
6. 先发送固定字符串 test，确认两块 C2 通信正常。
7. 再发送 HUMAN、RFID、ENV 等实际业务数据。
```

## 7. 功能设计

### 7.1 座位状态定义

系统将座位状态分为 6 类：

| 状态 | 含义 | 显示颜色建议 |
|---|---|---|
| EMPTY | 空闲，无人且未预约 | 绿色 |
| RESERVED | 已刷卡预约，但尚未入座 | 蓝色 |
| USING | 已签到并检测到有人使用 | 深蓝色 |
| AWAY | 已签到用户短时间离开 | 黄色 |
| OCCUPY | 临时离开超时，疑似占座 | 红色 |
| BAD_ENV | 当前温湿度不适合学习 | 橙色或白色提示 |

其中，OCCUPY 不再表示摄像头识别到物品，而是表示已绑定座位的用户长时间离开未归，系统将其判断为疑似占座或超时占用。

### 7.2 刷卡签到功能

S5 NFC/RFID 用于模拟学生身份识别。学生刷卡后，U1P 读取卡号，并通过 C2 无线链路上传到 U2P。U2P 根据当前座位状态判断该刷卡行为是预约、签到还是签退。

```text
学生刷卡
-> U1P 读取卡号
-> U1P 将卡号发送给本地 C2
-> C2 通过 ZigBee 转发到 U2P 侧 C2
-> U2P 接收卡号并绑定座位
-> E5 和手机管理页面同步更新状态
```

刷卡逻辑如下：

```text
空闲座位刷卡       -> 预约或签到
已签到座位再次刷卡 -> 签退并释放座位
管理员卡刷卡       -> 清除异常占座状态
```

### 7.3 人体感应检测功能

S7 人体感应模块用于判断座位前是否有人。

```text
S7 检测到人 -> human = 1
S7 未检测到人 -> human = 0
```

为了减少误判，系统可以加入简单防抖逻辑：

```text
human = 1 持续 3 秒以上 -> 判断为有人入座
human = 0 持续 5 秒以上 -> 判断为人员离开
```

这样可以避免人体感应信号短暂波动导致状态频繁切换。

### 7.4 临时离开与疑似占座判断

系统通过刷卡状态、人体感应结果和计时器判断座位状态。

```text
未刷卡 + 无人
-> EMPTY

刷卡后 + 尚未检测到人
-> RESERVED

刷卡后 + S7 检测到人
-> USING

USING 状态下 + S7 短时间检测不到人
-> AWAY

AWAY 超过设定时间仍无人
-> OCCUPY

用户再次刷卡
-> 签退并释放座位

管理员手机清除
-> 释放 OCCUPY 座位
```

实际演示和调试时，可以将离开超时时间设置为 20 秒到 60 秒，方便观察状态变化；正式设计中可以设置为 5 分钟或更长时间。

### 7.5 环境监测功能

S8 温湿度传感器由 U1P 读取，数据通过 C2 无线链路上传给 U2P。

环境判断规则建议如下：

```text
温度 < 18℃      -> 偏冷
温度 > 30℃      -> 偏热
湿度 > 75%RH    -> 潮湿
湿度 < 30%RH    -> 干燥
```

当环境异常时，E5 屏幕和手机管理页面均显示 BAD_ENV 提示，并显示当前温度和湿度。

### 7.6 手机端 HTTP 管理功能

U2P 通过网卡接入局域网，并运行 HTTP 服务。管理员手机连接同一局域网后，可以通过浏览器访问 U2P 的 IP 地址，查看座位状态并进行管理操作。

管理员手机端支持以下功能：

```text
1. 查看全部座位状态。
2. 查看某个座位的绑定卡号、当前状态和离开计时。
3. 查看当前温湿度和环境舒适度提示。
4. 清除 OCCUPY 疑似占座状态。
5. 强制释放异常座位。
6. 修改临时离开超时时间。
```

HTTP 管理功能不直接依赖 C2。C2 负责座位端数据无线传输，HTTP 服务由 U2P 通过网卡对管理员手机提供。

## 8. 通信协议设计

### 8.1 U1P 发送给 U2P 的业务数据

虽然中间通过 C2 ZigBee 传输，但业务层仍然采用一行一帧的文本协议，便于调试和展示。

人体感应数据：

```text
HUMAN,seat=A01,value=1
HUMAN,seat=A01,value=0
```

刷卡数据：

```text
RFID,seat=A01,card=20260001,event=scan
```

温湿度数据：

```text
ENV,temp=26.5,humi=48.2
```

心跳数据：

```text
HB,u1p=ok
```

### 8.2 U2P 发送给 U1P 的控制数据

U2P 可以通过反向链路向 U1P 发送控制指令，用于确认接收、请求数据或复位节点。

状态确认：

```text
ACK,rfid=20260001,result=ok
```

请求环境数据：

```text
GET,env
```

系统复位：

```text
CMD,reset
```

### 8.3 U2P 提供给管理员手机的 HTTP 接口

U2P 运行 HTTP 服务，建议接口设计如下：

```text
GET  /api/seats
获取全部座位状态

GET  /api/seats/A01
获取 A01 座位详情

POST /api/seats/A01/clear
清除 A01 的 OCCUPY 状态

POST /api/seats/A01/release
强制释放 A01 座位

GET  /api/env
获取当前温湿度数据

POST /api/config/timeout
修改临时离开超时时间
```

返回数据可采用 JSON 格式，例如：

```json
{
  "seat": "A01",
  "state": "USING",
  "card": "20260001",
  "human": 1,
  "away_seconds": 0,
  "temp": 26.5,
  "humi": 48.2
}
```

## 10. 状态机设计

单个座位的状态转换如下：

```text
EMPTY
  刷卡预约 -> RESERVED

RESERVED
  检测到人入座 -> USING
  超时未入座 -> EMPTY
  再次刷卡 -> EMPTY

USING
  人离开 -> AWAY
  再次刷卡 -> EMPTY

AWAY
  人返回 -> USING
  超时未归 -> OCCUPY
  再次刷卡 -> EMPTY

OCCUPY
  管理员手机 HTTP 清除 -> EMPTY
  E5 本地管理员清除 -> EMPTY
  用户刷卡确认 -> USING
```

状态判断重点不是单纯读取传感器，而是将刷卡身份、人体感应和时间规则结合起来，实现更接近真实自习室场景的状态管理。

## 11. E5 触摸屏界面设计

E5 触摸屏作为本地管理和显示界面，主要分为三个区域：

```text
左侧：座位地图
  显示 2 到 4 个座位，每个座位用不同颜色表示状态。

右侧：座位详情与环境监测
  显示座位编号、绑定卡号、当前状态、离开计时、温度和湿度。

底部：系统操作区
  显示刷新按钮、管理员清除按钮和网络状态提示。
```

触摸操作可以设计为：

```text
点击座位       -> 查看座位详情
点击管理员清除 -> 清除 OCCUPY 状态
点击刷新       -> 重新读取并刷新当前数据显示
```

界面中建议同时显示本机 IP 地址，例如：

```text
管理地址：http://192.168.1.100:8080
```

这样管理员可以直接根据 E5 屏幕上的地址，用手机浏览器访问管理页面。

## 12. 手机端管理页面设计

手机端页面由 U2P 提供 HTTP 服务，管理员通过浏览器访问，不需要单独安装 App。

页面建议包括以下内容：

```text
1. 页面标题：StudyGuard 座位管理后台
2. 座位列表：显示 A01、A02、A03、A04 等座位状态
3. 座位详情：显示绑定卡号、人体感应结果、离开时间和当前状态
4. 环境监测：显示温度、湿度和舒适度判断
5. 管理按钮：清除占座、释放座位、刷新数据
6. 配置区域：设置临时离开超时时间
```

页面可以使用原生 HTML + JavaScript 实现。前端通过 `fetch()` 请求 U2P 提供的 HTTP API，并根据返回 JSON 动态刷新座位状态。

## 14. 创新点

相比普通的座位检测系统，本项目有以下特点：

1. 使用 U1P + U2P 双主控架构，体现 MCU 底层采集与 Linux 上层管理的协同。
2. 加入两块 C2 ZigBee 子板，使座位端数据能够通过无线方式传输到管理端。
3. 采用 S7 人体感应、S5 刷卡识别和计时器融合判断，不是简单判断有人或无人。
4. 引入 NFC/RFID 身份识别，支持预约、签到、签退和管理员清除逻辑。
5. 使用 E5 触摸屏实现本地可视化座位地图，显示效果比简单指示灯更完整。
6. 在 U2P 上增加 HTTP 管理服务，使管理员手机可以通过网页远程查看和处理异常座位。
7. 融合温湿度环境数据，形成座位管理与学习环境监测一体化系统。
8. 状态机逻辑较完整，能够区分空闲、预约、使用中、临时离开和疑似占座。

## 17. 计划调整说明

相比原先只依赖 U1P 或 U1P 与 U2P 有线串口协同的方案，调整后的系统加入了 C2 ZigBee 子板和基于网卡的 HTTP 管理功能。U1P 仍然负责底层传感器数据采集，但采集结果不再只通过有线 UART 直接传给 U2P，而是先交给 U1P 侧 C2，再通过 ZigBee 无线链路发送到 U2P 侧 C2。U2P 接收到数据后，完成座位状态机判断、E5 触摸屏显示以及 HTTP 服务响应。管理员手机连接到同一局域网后，可以通过浏览器访问 U2P 的管理页面，实现查看座位状态、清除疑似占座和释放异常座位等操作。

这种调整使系统从单纯的本地座位显示终端，扩展为具备无线数据传输和移动端管理能力的智能座位管理系统。C2 的加入减少了 U1P 与 U2P 之间的布线依赖，也更符合物联网设备分布式采集与集中管理的设计思路；HTTP 管理功能则增强了管理员端的实际使用价值，使系统不仅能够显示状态，还能够通过手机进行远程管理。
